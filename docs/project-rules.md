# Intem Studio 프로젝트 규칙

이 저장소에서 작업할 때 따르는 **공통 규칙**입니다. 공개 사이트에는 게시되지 않습니다 (`docs/`는 Jekyll에서 제외).

- 에이전트 진입점: [../AGENTS.md](../AGENTS.md)
- 블로그 글 작성: [writing-guide.md](writing-guide.md)
- 커밋·저장소 관리: 이 문서

---

## 커밋 메시지

**모든 커밋의 제목과 본문은 한글로 작성합니다.**

영문 파일명·API명·패키지명은 그대로 써도 됩니다. 설명 문장은 한글입니다.

### 제목 (첫 줄)

- 변경 내용을 **한 문장**으로 요약합니다.
- 권장 길이: **50자 전후** (Git UI에서 잘리지 않게).
- 문체: **명령형** 또는 **과거형** 중 하나로 팀 내 일관되게 씁니다.  
  예: `Unity 에디터 로그 글 추가` / `Unity 에디터 로그 글을 추가했다`

### 본문 (선택)

- **왜** 바꿨는지, **범위**가 무엇인지 1–3문장으로 씁니다.
- 여러 논리 단위로 나눠 커밋할 때는 커밋마다 본문에 역할을 한 줄 적습니다.

### 예시

**좋은 예**

```
제목: Unity 에디터 로그 글과 P1–P3 샘플 코드 추가

본문: Log 래퍼 세대별 진화와 컴파일 제거 설계를 정리했다.
unity 인덱스에 페이지 목록을 반영하고, code/Log/에 참고 구현을 둔다.
```

```
제목: 내부 작성 가이드와 system-evolution 템플릿 추가

본문: 블로그 글 공통 규칙과 진화형 설계 전용 템플릿을 docs/에 둔다.
contributor 문서가 사이트에 올라가지 않도록 Jekyll exclude에 docs/를 넣는다.
```

**피할 예**

- 제목·본문이 영문인 커밋 (`Add editor log article`)
- 제목만 있고 무엇을/왜 바꿨는지 알 수 없는 경우 (`수정`, `업데이트`, `fix`)

### 커밋 나누기

- **사이트에 올라가는 콘텐츠**와 **내부 문서·설정**은 가능하면 커밋을 나눕니다.
- 한 커밋에 서로 관련 없는 변경을 섞지 않습니다.

## 규칙 추가

새 규칙이 생기면 이 문서에 절을 추가하고, [AGENTS.md](../AGENTS.md)에서 링크를 갱신합니다. Cursor 전용 동작(예: `alwaysApply`)이 필요하면 `.cursor/rules/`에 `.mdc` 파일을 둡니다.
