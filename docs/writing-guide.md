# Intem Studio 개발 블로그 글쓰기 가이드

이 문서는 **저장소 내부용** 작성 규칙입니다. 공개 사이트에는 게시되지 않습니다.

글 유형마다 **전용 템플릿**을 두고, 새로운 목적의 글이 생기면 `docs/writing/templates/`에 템플릿을 추가해 나갑니다. 모든 글은 아래 **공통 규칙**을 따른 뒤, 해당 유형의 템플릿을 적용합니다.

**관련 규칙:** [project-rules.md](project-rules.md) — 커밋 메시지 등 저장소 공통 규칙 · [AGENTS.md](../AGENTS.md) — 에이전트 진입점  
**참고 사례:** [unity/editor-log.md](../unity/editor-log.md) — 시스템 진화형 설계 글

## 1. 글의 목적·독자 (작성 전 체크리스트)

발행 전에 아래를 먼저 정합니다. 독자에게 “목적/독자” 박스를 넣을 필요는 없고, **도입 문단에 녹입니다.**

| 항목 | 질문 예시 |
|------|-----------|
| **목적** | 이 글이 독자에게 무엇을 알려 주는가? |
| **독자** | 누가 읽는가? (기본: **같이 고민하는 엔지니어**) |
| **범위** | 무엇을 다루고, 무엇은 다루지 않는가? |
| **핵심 통찰** | 독자가 가져갈 메시지 1–2개는? |

예시 (내부 메모):

> 목적: Unity에서 직접 로그 래퍼를 설계하는 개발자에게, 컴파일 제거와 태그 필터 설계를 설명합니다.  
> 독자: C#/Unity에 익숙한 게임 클라이언트 엔지니어.  
> 범위 밖: 프로덕션 로그 수집·분석 파이프라인 전체.

Intem Studio는 **케이스 소스**로만 언급합니다. 회사 홍보나 1인칭 스토리텔링보다 **재현 가능한 설계 설명**을 우선합니다.

## 2. 공통 문체·톤

- 기본은 **합니다체**의 중립적인 기술 문체입니다. 해체(「~한다」)나 반말과 **섞지 않습니다**.  
  예: “P3는 출력 API 전부에 Conditional을 적용합니다.”
- 1인칭(“나는/우리는”)은 **교훈·마무리**에만 가끔 허용합니다. 이때도 어미는 합니다체를 유지합니다.  
  예: “P2에서는 런타임 방어만으로는 호출부 비용이 남는다는 것을 배웠습니다.”
- 감성·강조 표현은 **도입·마무리**에만 넣고, 본문은 명확한 기술 서술을 유지합니다.
- 코드 설명은 주석 나열보다 **문장**으로 씁니다.  
  예: “`defaultEnabled`는 에디터에서만 true이며, 플레이어 빌드에서는 false입니다.”

## 3. 공통 형식·마크다운

### 단락 vs 리스트

- 본문 설명은 **짧은 단락** 위주로 씁니다.
- 아래는 **리스트 사용을 허용**합니다.
  - API·설정 항목 나열
  - 비용·체크 항목 목록
  - **실무 가이드라인**(규칙·체크리스트)
  - 짧은 설계 원칙 bullet (3–4개 이내)

### 표

축이 명확할 때만 사용합니다.

- 세대별·버전별 비교
- 레벨·태그·옵션 목록
- “컴파일 vs 런타임” 같이 시점·결과가 대비되는 비교

### 코드 블록

- 실제 구현에서 가져온 스니펫만 사용합니다.
- **이 글에서 이해에 필요한 최소 길이**만 포함합니다.
- 코드 블록 뒤에는 **1–2문장** 설명을 붙입니다.
- 레포 파일에서 발췌했으면 **소스 코드 섹션**에 “발췌”임을 명시합니다.

### 시각 보조

- 진화·흐름은 ASCII 다이어그램을 허용합니다.  
  예: `P1 ──기능 확장──▶ P2 ──역할 분리──▶ P3`
- 긴 글은 절·표 간 **자기 참조 링크**를 사용합니다.  
  예: “위 표”, “앞 절”, “소스 코드 절”
- 한글 제목 앵커는 배포 환경(GitHub Pages)에서 깨질 수 있으므로, **배포 후 링크를 확인**합니다.

### 대목차·소제목

- 상위 섹션은 보통 **4–8개 대목차**입니다.
- 한 섹션이 길면 **2–3단계 소제목**으로 나눕니다. (레벨·태그·API 등)

## 4. 공통 기술·레퍼런스

### 출처 우선순위

1. **공식 문서** — Microsoft Docs, Unity Manual, Unity Scripting API, Unity Package 문서
2. **1차 자료** — 레포 코드, 스펙, 이슈
3. **보조 자료** — 블로그, Stack Overflow (공식 문서 보완용)

개념 설명(예: `ConditionalAttribute`) 뒤에는 가능하면 **공식 문서 링크를 하나 이상** 붙입니다. Unity 관련 글은 Unity 문서를 **최소 한 번** 인용합니다.

### 레포 코드 인용

- **소스 코드** 섹션을 두고, GitHub blob/tree 링크를 제공합니다.
- 스니펫이 전체 파일의 일부임을 명시합니다.
- repo에 없는 의존 타입·에셋이 있으면 **솔직히 고지**합니다.

## 5. Jekyll 발행 형식

`unity/` 페이지 또는 `_posts/` 글에 공통으로 적용합니다.

```yaml
---
layout: page          # 또는 post
title: "기술 요소 — 맥락"
permalink: /unity/슬러그/
tags:
  - unity
  - logging          # 주제별 자유 추가
categories: unity     # _posts 사용 시 unity 인덱스 노출용
---
```

- **`tags`**: 나중에 주제별로 찾기 좋게 3–5개 정도.
- **`permalink`**: 영문 슬러그 권장 (앵커·URL 안정성).
- **`categories: unity`**: `_posts` 글은 [unity/index.md](../unity/index.md) 목록에 포함됩니다.
- `unity/` 하위 `.md` 페이지는 경로 기준으로 unity 인덱스에 자동 포함됩니다.
