## 권장 대목차 순서

시간순이 아니라 **독자가 지금 쓰는 설계(최신 세대)부터** 소개한 뒤, 과거로 내려갑니다.

| 순서 | 섹션 | 내용 |
|------|------|------|
| 1 | **도입** | 문제 한 줄 · 세대·현재 버전 소개 · 사례 출처(레포) |
| 2 | **현재 설계가 하는 일** | 최신 세대의 역할·API·설정 vs 출력 구분 |
| 3 | **세대 개요** | 짧은 표 또는 ASCII로 P1→P2→P3 방향 요약 |
| 4 | **과거 세대에서 배운 점** | 구체적 실패·비용·오해 (코드·표·프로파일) |
| 5 | **현재 설계 원칙** | 핵심 원칙 3–4개, 각각 짧은 예시·코드 |
| 6 | **비교 요약** | 세대별 표 (백엔드, 빌드 비용, 필터, API 형태 등) |
| 7 | **실무 가이드라인** | 번호 목록 — 언제 쓰고, 언제 쓰지 않는지 |
| 8 | **마무리** | 원칙·교훈 1–2단락 (1인칭 허용) |
| 9 | **소스 코드** | repo 경로, 발췌 고지, 의존 타입 고지 |
| 10 | **참고** | 공식 문서 링크 |

§2「현재 설계」가 길면 **로그 레벨, 태그, 호출 형태, 설정 API**처럼 소제목으로 나눕니다.

## 섹션별 작성 가이드

### 1. 도입 (1단락 + 필요 시 구분선)

- **문제 한 줄** — 독자가 공감할 리스크·비용
- **글 범위** — 세대 이름, 현재 버전
- **사례 출처** — “Intem Studio 프로젝트에서 가져왔다” 수준. 홍보 톤은 피함

예시 톤:

> Unity에서 개발용 `Log` 래퍼를 직접 설계하는 경우가 많습니다. 편리하지만 잘못 두면 개발 편의 코드가 릴리스 빌드 성능까지 갉아먹습니다. 이 글은 로그를 세 번 다시 짜며 얻은 원칙을 정리합니다. 구현은 **P1 → P2 → P3**(현재)로 부릅니다.

### 2. 현재 설계가 하는 일

- 이 시스템이 **무엇을 하고, 무엇을 하지 않는지** 먼저 밝힙니다.
- **출력 API** vs **설정·조회 API**처럼 역할이 갈리면 구분해 설명합니다.
- 대표 API 호출 예시 2–3줄.
- 전체 구현 파일 링크 + “스니펫은 발췌” 한 줄.
- 마지막에 “이후 절에서는 세대별 진화를 다룬다”로 다음 섹션 연결.

### 3. 세대 개요

- 3–4행 표 또는 ASCII 흐름도.
- 각 세대를 **한 줄 방향성**으로만 요약 (상세는 §4·§5에서).

```text
P1  ──기능 확장──▶  P2  ──역할 분리·단순화──▶  P3
```

### 4. 과거 세대에서 배운 점

- **한 세대(보통 문제가 드러난 세대)**에 초점을 맞춰도 됩니다. (editor-log는 P2 중심)
- 반드시 넣을 것:
  - 잘못된 가정 (“early return이면 충분하다” 등)
  - **구체적 비용** (인자 평가, GC, 핫 패스)
  - 짧은 **잘못된 호출 예시** 코드
- 컴파일 vs 런타임처럼 축이 명확하면 **표** + 표 아래 1–2문장 보강.
- 이전 세대의 부분적 성공(예: P1 Conditional)도 공정하게 언급합니다.

### 5. 현재 설계 원칙

- 원칙 **3–4개**. 각 원칙 = 소제목 + 2–4단락 + (필요 시) 코드 스니펫.
- 첫 원칙 앞에 **한 줄 요약 인용**을 두면 좋습니다.  
  예: `> 로그 출력은 에디터 전용. 릴리스 메시지는 Debug.Log* 등 별도 경로.`
- “왜 이전 세대와 다른 선택을 했는지”를 원칙마다 짧게 연결합니다.

### 6. 비교 요약

- 항목 × 세대 표 하나로 마무리.
- §4의 컴파일/런타임 표와 **중복되면 역할을 나눕니다.**  
  - §4: 개념·메커니즘 비교  
  - §6: 제품·설계 선택 비교

### 7. 실무 가이드라인

- **번호 목록** 4–6개.
- “~할 때 → ~한다” 형태.
- 독자가 내일 코드에 바로 적용할 규칙.

### 8. 마무리

- 설계가 다루는 **축**(컴파일, 필터, 호출부 비용 등)을 한 번 더 묶습니다.
- 세대별 한 줄 회고. 여기서만 “배웠다” 같은 1인칭 허용.

### 9. 소스 코드

- tree 링크 + 세대별 파일 표.
- repo에 없는 의존성 명시.

### 10. 참고

- Microsoft Docs, Unity Manual/API, Package 문서 위주.
- 대안(구조화 로그 등)이 있으면 짧게 링크.
